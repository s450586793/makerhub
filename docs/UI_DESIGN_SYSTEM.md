# MakerHub UI 设计系统

这份文档记录 MakerHub 当前的新 UI 风格。后续 Codex 修改前端时，先读本文件，再读对应页面代码。这里的规则优先级高于旧的浅色 SaaS 风格规则。

## 总体气质

- MakerHub 是“真实归档数据工作台”，不是营销页，也不是大留白落地页。
- 首屏应该直接是可操作的工作台：列表、筛选、状态、任务、导入、设置等。
- 视觉目标是深色、克制、密集但清楚，适合长时间管理模型库。
- UI 文案以功能为主，不在页面里解释“如何使用这个界面”的说明长文。
- 页面顶部、卡片、表格、弹窗都要优先服务扫描、比较、批量操作。

## 主题和颜色

- 默认设计以当前深色模式为第一视觉基准，同时保持浅色模式可用。
- 主题必须支持 `light`、`dark`、`auto`。
- 核心 token 在 `app/static/css/app.css`：
  - 背景：`--bg`、`--bg-accent`
  - 表面：`--surface`、`--surface-soft`、`--surface-strong`、`--surface-floating`
  - 边框：`--border`、`--border-strong`
  - 文字：`--text`、`--muted`
  - 强调：`--brand`、`--brand-strong`、`--brand-soft`
- 深色模式主背景接近 `#111412`，卡片表面接近 `#171b18` / `#1e2420` / `#252b27`。
- 主强调色是绿色，不要改成蓝紫、紫色渐变、彩虹色或霓虹色。
- 状态色只在必要时使用：成功绿色、警告橙色、错误红色。
- 不要硬编码白色线条、白色背景或浅色阴影；深色模式必须通过 token 派生。

## 布局

- App 采用左侧固定导航 + 右侧内容工作区。
- 左侧导航是品牌、主导航、用户、版本的工作台侧栏；不要做成顶部营销导航。
- 主内容宽度优先使用现有 `--content-width` 和页面容器，不要随意拉满到不可读。
- 首页、模型库、订阅库、本地库、源端刷新、归档任务、设置、日志的顶部功能栏要保持同一高度和密度。
- 常规顶部功能栏使用 `app-page-toolbar` 或 `library-toolbar` 这类紧凑工具栏：
  - 高度约 82-84px。
  - 标题约 22px。
  - 统计卡片约 46px 高。
  - 操作按钮、筛选、状态信息尽量并排，移动端再折行。
- 不要新增大 hero、宣传卡、装饰插画或渐变背景。

## 卡片和分组

- 卡片只用于独立条目、工具面板、弹窗内容和重复列表项。
- 不要把页面 section 全部做成一层套一层的大卡片。
- 不要卡片里再套一堆小卡片，尤其在 3MF 列表、文件列表、设置表单里要避免碎片化。
- 3MF/文件列表在深色模式下应像紧凑列表或表格，不要每一项都是重边框小块。
- 常规圆角使用 6px 或 8px；侧栏大容器可以更大，但业务卡片不要动辄 16px+。
- 边框优先使用 `--border`，背景优先使用 `--surface*` token。
- 阴影要很克制；深色模式下更多依赖层级色和边框，不靠大阴影。

## 按钮和操作

- 主按钮使用绿色强调，次按钮使用克制描边/表面色。
- 危险操作使用红色，但不要大面积红色。
- 图像轮播、详情页主图左右切换等覆盖在媒体上的按钮应透明、小、悬浮，不要挡住图片。
- 有图标库时优先用现有图标；不要用 emoji 当功能图标。
- 按钮文字要短，避免撑破布局。例如“上传中”优于“正在上传到服务器”。
- 复制、下载、打开、刷新、清除、删除等动作必须有 disabled/loading/failure 状态。

## 表单和设置页

- 设置页按功能分 tab。当前主要包括：系统、线上账号、代理、本地整理、Token、模型分享、高级、用户、通知。
- “线上账号”和“代理”分开；HTTP 代理不放在线上账号里。
- 添加账号用弹窗完成平台、账号、密码输入；账号列表在页面内展示状态和操作。
- Token 管理要像表格：名称、过期时间、权限、Token 字符、操作横向列出。
- 分享设置默认内容、公开访问地址、默认有效期等属于设置项，不应每次点击分享都弹选择项。
- 表单要密集但清楚，字段说明短，不要一屏堆大段解释。

## 模型库和详情页

- 模型库卡片要便于扫图、扫标题、扫来源状态。
- 卡片封面区域稳定，不因徽标、标题、hover 或按钮导致布局跳动。
- 本地库、订阅库、模型库的多选/批量操作尽量保持同一套交互。
- 详情页左侧主图和图集、右侧文件/3MF/采集信息等区域要对齐。
- 右侧文件列表高度应和左侧图片区域协调，多则滚动，少则留空，不要强制固定条数。
- 本地导入模型不展示评论板块。
- 3D 预览应是点击后加载的弹窗/轻量预览，不影响模型列表性能。

## 本地导入和任务进度

- 本地整理进度是工作台状态位，不是上传弹窗里孤立的进度条。
- Web 导入、iOS 导入、local 文件夹导入都应反映到同一个本地整理进度展示。
- 上传阶段和后台整理阶段应合并到用户可理解的一条进度语义里。
- 整理完成显示空闲；失败和跳过要能展开查看原因。
- 长任务必须后台运行，前端只读进度，不能阻塞页面交互。

## 深色模式硬规则

- 新增 UI 必须同时检查深色模式和浅色模式。
- 不得出现白色分割线、白底输入框、黑字暗底、浅色图标不可见。
- 不要使用只适合浅色模式的图片/边框/阴影。
- 使用 `color-mix` 时要确认深色模式下对比度足够。
- 任何截图拼图、快照、生成图如果有背景线条，深色模式下也要匹配主题。

## 响应式

- 移动端可以折行，但不能让文字互相覆盖或按钮超出容器。
- 不用 viewport width 缩放字体。
- 固定格式组件要有稳定尺寸：工具栏统计、进度条、模型卡片封面、图册、文件列表、icon button。
- 长标题和长文件名必须省略或换行，不能撑坏布局。

## 禁止模式

- 不要做营销 landing page。
- 不要添加蓝紫渐变、渐变球、光斑、玻璃拟态装饰。
- 不要把所有内容做成一堆浮动大卡片。
- 不要用内联颜色/间距替代设计 token。
- 不要为了“好看”牺牲信息密度和操作效率。

## 修改前检查

改 UI 前至少确认：

1. 这次改动属于哪个页面/模块，先读 `docs/modules/frontend_settings.md`。
2. 是否复用 `app/static/css/app.css` 和 `frontend/src/style.css` 现有 token/组件。
3. 深色模式和浅色模式是否都可读。
4. 顶部工具栏高度是否与其它页面一致。
5. 文本、按钮、文件名、进度状态是否会在移动端溢出。
6. 是否需要跑 `npm --prefix frontend run build`。
