# 首页浏览器验证重试设计

## 背景

首页的平台状态卡会在 `3MF` 下载被网页验证拦截时显示“需要浏览器确认”和“打开官网”。用户完成浏览器验证后，还需要进入其他页面或等待后续流程才能恢复被阻塞的归档任务。

后端已经提供 `POST /api/tasks/missing-3mf/verification-verified`：它标记指定平台已完成验证，并在后台重试该平台因验证阻塞的 `3MF` 任务。首页卡片尚未暴露这个明确操作。

## 目标

- 在首页“需要浏览器确认”卡片中增加 `已验证` 操作。
- 用户点击后立即调用既有验证确认接口，并后台恢复同平台受验证阻塞的 `3MF` 归档任务。
- 卡片在请求期间防止重复提交，并显示已有的提交中/结果反馈。

## 交互

- 仅当状态卡的 `state` 为 `verification_required` 时显示两个操作：`打开官网` 和 `已验证`。
- `打开官网` 保持为外链，在新标签页中打开对应平台主页。
- `已验证` 是紧凑的主按钮，不弹二次确认；用户应只在已完成浏览器验证后点击。
- 点击后按钮显示“提交中”，请求成功后展示接口返回的后台重试提示，并刷新首页状态。
- `cookie_invalid`、`daily_limit`、`network_error`、`checking` 和可归档状态不得显示 `已验证`。

## 数据流

1. `account_health.snapshot_to_source_card` 在 `verification_required` 时输出显式操作数组。
2. 操作数组包含外链主页操作，以及 `POST /api/tasks/missing-3mf/verification-verified` 的 API 操作，body 为 `{ "platform": "cn" | "global" }`。
3. 首页继续复用 `dashboardStatusActions` 和 `runStatusAction` 渲染及执行 API 操作。
4. 既有接口调用 `mark_account_ok`，并将同平台验证阻塞的 `3MF` 重试放入后台任务；状态事件驱动首页刷新。

## 错误处理

- 请求失败时保留原有“需要浏览器确认”状态，显示 API 返回的错误，不伪造已恢复状态。
- 后台任务只代表已接受；后续下载结果仍由归档队列和账号 health 探针决定。
- 不依据中文文案或前端推测增加按钮，确保状态契约由后端单点定义。

## 测试

- 后端：验证状态卡仅输出正确平台的 `已验证` API 操作；其他状态不输出该操作。
- 前端：`dashboardStatusActions` 保留显式 API 操作，并对验证卡输出 `打开官网` 与 `已验证`。
- 回归：既有验证确认接口仍标记账号可用并安排后台重试。
- 运行相关 Python 测试、前端测试和生产构建。

## 非目标

- 不自动绕过或自动完成网页验证。
- 不把“已验证”用于重新登录、下载限额或网络异常。
- 不改变 Cookie、指纹浏览器或归档队列的底层执行逻辑。
