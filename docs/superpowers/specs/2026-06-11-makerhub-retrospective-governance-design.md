# MakerHub 重度复盘治理设计

日期：2026-06-11
状态：设计已确认，待实现计划

## 背景

MakerHub 近期反复出现同类问题：首页源站状态误报验证、历史 `missing_3mf` 残留影响当前状态、源端刷新和归档任务边界混乱、日志与任务状态互相干扰、线上现象和代码判断不一致。

这不是单个 bug，而是状态定义、流程 ownership、线上排查和发布验证没有形成闭环。后续目标不是继续增加复杂能力，而是把 MakerHub 收束成稳定的真实归档数据工作台。

## 复盘目标

1. 状态不再混用。首页只展示当前源站账号是否可用于归档，不被历史失败、旧 `missing_3mf`、老任务残留或日志里的旧验证事件污染。
2. 流程不再互相拖累。源端刷新、归档下载、订阅同步、库信息补全、缺失 3MF 重试要有明确边界。
3. 线上判断可以落地。遇到“首页提示需要验证，但网页能下载”时，必须用线上 Cookie 探针确认真实原因，而不是靠日志关键词推断。

## 非目标

- 不重新设计 UI。
- 不引入新的浏览器自动化验证方案。
- 不继续做验证码滑块映射、内嵌验证窗口或 CloakBrowser 类方案。
- 不扩展复杂反爬能力。

MakerHub 的方向是：用户手动在官网完成验证，系统负责正确识别、记录和恢复状态。

## 状态契约

状态必须分为 4 类，不能互相推导。

### 账号可用状态

表示当前配置的源站 Cookie / 账号是否能执行当前动作。它是首页源站卡片唯一可信来源。

允许状态：

- `ok`：当前可用。
- `verification_required`：当前线上 Cookie 触发验证页。
- `daily_limit`：当前账号触发每日上限。
- `cookie_invalid`：Cookie 失效或登录态不可用。
- `network_error`：网络或上游访问异常。
- `unknown`：尚未探测或证据不足。

允许写入来源：

- 当前轻量下载探针。
- 当前归档下载结果。
- 当前缺失 3MF 重试结果。
- 当前任务中被明确分类的验证 / 上限 / Cookie 异常。

禁止写入来源：

- 历史 `missing_3mf` 列表。
- 旧任务失败记录。
- 旧日志事件。
- 普通 `403` 字符串匹配。
- scrapling trace 的单条失败日志。

### 任务执行状态

表示单个归档、源端刷新、订阅同步、补全或重试任务的生命周期。

典型状态：

- `pending`
- `running`
- `completed`
- `failed`
- `paused`
- `cancelled`

任务失败不能直接等于账号异常。只有失败原因被明确归类为验证、Cookie 或每日上限时，才能通过账号健康 service 更新账号可用状态。

### 失败明细状态

表示具体失败条目，例如 `missing_3mf`、下载失败记录、历史归档错误。

这些状态只用于：

- 归档页面。
- 重试入口。
- 错误统计。
- 具体模型排障。

这些状态不能影响首页源站卡片。历史失败即使包含“验证”字样，也只是历史事实，不代表当前账号仍然需要验证。

### 诊断事件状态

日志、探针记录、scrapling trace、HTTP 403 记录都是证据，不是最终状态。

诊断事件可以解释原因，但 UI 不能直接把它们当成当前账号状态展示。

## 首页源站状态规则

首页源站卡片只读取账号健康快照，不读取任务表、失败表、日志表或 `missing_3mf` 历史列表。

硬规则：

- 历史失败不影响首页。
- 历史验证事件不影响首页。
- 当前归档下载成功后，对应源站账号状态可以恢复为 `ok`。
- 当前缺失 3MF 重试成功后，对应源站账号状态可以恢复为 `ok`。
- 用户在电脑浏览器手动过验证后，MakerHub 不能假设线上 Cookie 已同步，必须用线上 Cookie 探针确认。
- `403` 不能默认等于验证码，必须分类为验证页、每日上限、Cookie 失效、普通拒绝、页面结构变化或网络问题。

## 流程边界

### 源端刷新

只负责从订阅源、作者页、合集、收藏夹发现模型，并更新源端模型索引。

它不负责：

- 下载 3MF。
- 补全本地归档。
- 处理历史缺失 3MF。
- 直接改变首页账号状态。

只有刷新过程中明确触发当前 Cookie 验证失败，并经账号健康分类确认后，才可以更新账号健康快照。

### 归档下载

只负责把指定模型下载到本地库。

它可以更新账号健康状态，因为下载动作最接近真实可用性：

- 下载成功：对应源站可以恢复为 `ok`。
- 遇到验证页：写 `verification_required`。
- 遇到每日上限：写 `daily_limit`。
- Cookie 失效：写 `cookie_invalid`。
- 网络错误：写 `network_error`。

### 缺失 3MF 重试

只负责处理历史缺失文件。

规则：

- 重试成功：清理或关闭对应失败明细，并把对应源站当前账号状态恢复为 `ok`。
- 重试失败：只更新该失败明细。
- 历史失败不能继续污染首页。

### 订阅同步

只负责同步订阅库中的作者、合集、收藏夹列表和元数据。

订阅同步失败不等于归档不可用，也不等于需要验证。只有当同步接口明确返回登录失效或验证页时，才写诊断事件，并由账号健康探针二次确认。

### 库信息补全 / 索引重建

只负责本地数据一致性，例如 `meta.json`、数据库索引、作者信息、封面和统计字段。

这类后台任务不能影响首页源站状态，也不能阻塞归档主任务。索引陈旧最多显示“后台重建中”，不应反复刷 `WARN` 干扰判断。

## 复杂度削减原则

### 保留

- Cookie 方式：用户自己登录 MW，MakerHub 使用配置的 Cookie 做下载和同步。
- 外跳官网 / 模型页：需要验证时让用户去官网处理。
- 轻量探针：用线上 Cookie 检查当前账号能否访问 / 下载。
- 独立队列：源端刷新、归档下载、缺失重试、补全任务各自排队。
- 批量落库和批量刷新：任务过程中少写日志、少刷新前端，完成后统一更新。

### 降级

- scrapling / HTML 获取 trace：只作为 fallback 证据，不主导首页状态。
- 日志中心：普通用户只看排障价值，AI / 开发者可以查详细日志。
- 自动元数据补全：失败时后台慢慢补，不影响归档下载主流程。
- 实时状态刷新：降级为批次快照和必要轻量轮询。

### 删除或不再投入

- 内嵌浏览器验证窗口。
- 验证码滑块映射到 MakerHub。
- CloakBrowser / 浏览器组件类依赖，除非存在无法替代的运行时硬依赖。
- 把历史错误当成当前状态的推导逻辑。
- 一个任务同时做刷新、下载、补全、重试和状态判定的混合流程。

新功能准入问题：

1. 能不能让归档更稳定？
2. 能不能让线上问题更好判断？
3. 会不会让状态来源更混乱？

如果前两条没有明显收益，第三条有风险，则不做。

## 线上排查规范

线上问题按固定顺序排查。

### 1. 查看当前账号健康快照

确认首页读取字段：

- `status`
- `reason`
- `source`
- `updated_at`
- `site`

先判断状态是不是新鲜，来源是不是当前探针或当前任务。

### 2. 用线上 Cookie 做轻量探针

随机选一个低风险模型或固定测试模型，触发只验证下载入口的探针。

探针必须输出明确分类：

- 下载入口可达。
- 验证页。
- 每日上限。
- Cookie 失效。
- 网络错误。
- 页面结构变化。

### 3. 查看当前任务

只看当前运行中的归档、源端刷新、缺失重试任务。

历史 `missing_3mf`、旧日志、旧失败记录只能作为背景，不作为当前账号状态依据。

### 4. 查看详细日志

日志用于解释原因，不用于直接改 UI 状态。

如果日志里有大量 `403`，必须继续确认是验证、上限、Cookie、请求头、页面结构还是普通访问拒绝。

## 发布守门

### 测试要求

改状态逻辑时，必须至少运行：

```bash
.venv/bin/python -m unittest tests.test_source_health
```

改归档 / 缺失重试时，必须至少运行：

```bash
.venv/bin/python -m unittest tests.test_missing_3mf tests.test_source_library
```

改源端刷新时，必须运行 source refresh / worker / queue 相关测试。

对线上误报类修复，必须有回归测试表达：历史失败不影响当前首页状态。

### 推送要求

- 推送前必须查看 `git diff`。
- 只暂存当前任务直接产生的文件，禁止 `git add .`。
- 不混入无关文件或其他工具生成物。
- 只有用户明确要求推送时才 bump 版本号、更新 `CHANGELOG` / README 并 push。
- README 只直接展示最新三条 release，旧记录放折叠区。

## 后续实现方向

这份复盘规格后续应拆成实现计划，而不是一次性盲改。

建议实现顺序：

1. 账号健康快照 service 与字段收口。
2. 首页源站卡片只读账号健康快照。
3. 归档下载 / 缺失重试成功后恢复账号状态。
4. 403 / 验证 / 上限 / Cookie 失效分类收口。
5. 源端刷新、归档下载、补全、订阅同步的队列边界复查。
6. 线上探针命令或诊断入口固化。
7. 回归测试和发布守门补齐。

## 验收标准

- 首页状态不会被历史 `missing_3mf`、旧日志或旧任务失败污染。
- 使用线上 Cookie 探针可以解释“网页能下但 MakerHub 提示验证”的真实原因。
- 当前归档或重试成功后，对应源站状态能恢复为正常。
- 源端刷新卡住不会让归档下载、首页状态或订阅库一起卡住。
- 状态、任务、失败明细、诊断事件在代码中有清晰 service 边界。
- 对关键误报问题有自动化回归测试。
