# 账号归档状态收敛设计

## 背景

线上账号页面同时展示了来源同步、Cookie 探针和指纹浏览器关联三种独立状态。当前设置页会在资料或来源同步仍可读取时显示“正常”，而首页会依据 `account_health` 中的 `cookie_invalid` 阻断 `3MF` 下载。用户会看到同一国内账号既“正常”又“Cookie 异常”。

## 目标

每个平台账号只展示一个面向操作的主状态，回答“现在能否归档下载”。资料同步和浏览器 profile 都保留，但不再与主状态竞争，也不把可选浏览器能力显示为故障。

## 状态模型

主状态由平台 `account_health` 决定，并在首页、设置页和归档操作中使用同一优先级：

| 原始状态 | 主状态 | 样式 | 主操作 |
| --- | --- | --- | --- |
| `ok` 或已确认当前下载成功 | 可归档 | 绿色 | 无 |
| `auth_required`、`cookie_invalid` | 需要重新登录 | 红色 | 重新登录 |
| `verification_required`、`cloudflare` | 需要浏览器确认 | 橙色 | 打开浏览器 |
| `download_limited` | 今日下载受限 | 橙色 | 无 |
| `http_error`、`html_response`、未知 | 状态待确认 | 中性或橙色 | 测试 |

设置页不得再用“资料或来源同步可读取”覆盖 `auth_required`、`cookie_invalid`、验证拦截或下载限制。来源同步成功只能说明资料已更新，不能说明 `3MF` 下载可用。

## 页面表现

### 设置 - 线上账号

- 账号标题只显示上述主状态，不再显示泛化的“正常”。
- 主状态下方显示简短的归档结论，例如“国内站 `3MF` 下载需要重新登录”。
- “资料同步：07/14 01:28”和关注/收藏统计保留为辅助信息。
- `browser_profile_id` 缺失时不显示“浏览器未关联”提示；账号可归档时不显示浏览器相关说明。
- 只有主状态为“需要浏览器确认”或用户主动点击“打开浏览器”时，才展示浏览器关联、同步和恢复结果。
- Cookie 已失效时，“重新登录”是唯一强调操作；“测试”保留为次操作。浏览器操作不抢占主操作。

### 首页 - 源端状态

- 保持以 `account_health` 为依据的可归档结论。
- 文案与设置页主状态使用同一映射：`Cookie 异常` 改为“需要重新登录”，`需要验证` 改为“需要浏览器确认”。
- 首页不展示来源同步统计或浏览器关联状态。

## 数据流

- 后端继续保存来源同步信息、浏览器会话信息与 `account_health` 原始快照，不删除诊断数据。
- 配置接口向线上账号条目补充经过统一映射的 `operational_status`、`operational_message` 与 `operational_action`。前端不再自行用来源同步结果降级认证失败。
- 首页源端卡片使用相同映射函数，避免分别维护中文标签和优先级。
- 归档 Worker 的 gate 保持原逻辑；本次仅让 UI 显示与 gate 一致的结论。

## 测试

- 后端：`cookie_invalid`、`auth_required`、验证、限额、未知和可归档状态的 API 映射。
- 前端：设置页状态不可被来源同步成功覆盖；浏览器未关联在可归档状态下不显示；主操作与主状态匹配。
- 回归：首页和设置页对同一原始状态产生一致标签。

## 非目标

- 不改变 Cookie 探针、归档下载、来源同步或指纹浏览器的实际执行逻辑。
- 不删除浏览器登录和手动同步能力。
